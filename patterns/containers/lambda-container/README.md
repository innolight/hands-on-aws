# lambda-container

```
Client
  │  HTTPS
  ▼
Lambda Function URL
  │  invoke event (path, headers, body)
  ▼
Lambda Web Adapter (LWA)
  │  HTTP proxy → localhost:3000
  ▼
Express :3000
  ├── GET /health  → 200 {"status":"ok"}   (no auth)
  └── GET /quote   → 200 {"quote":...}     (x-api-key required)
```

- [**Lambda Function URL**](https://docs.aws.amazon.com/lambda/latest/dg/lambda-urls.html) — free HTTPS endpoint attached directly to the function; `AuthType: NONE` (app handles auth via `x-api-key` header)
- [**Lambda Web Adapter (LWA)**](https://github.com/awslabs/aws-lambda-web-adapter) — AWS open-source binary injected as a Lambda extension; intercepts the runtime bootstrap, starts Express, then proxies Lambda invoke events to `localhost:3000`
- **ECR image** — ARM64 container built from the `lambda` stage of `example-container/Dockerfile`; pushed as tag `:lambda` (distinct from `:latest` used by ECS patterns in the same ECR repository)
- [**Secrets Manager**](https://docs.aws.amazon.com/secretsmanager/latest/userguide/intro.html) — API key auto-generated by CDK; injected as `API_KEY` env var via CloudFormation dynamic reference (`{{resolve:secretsmanager:...}}`). SSM SecureString dynamic references are not supported in Lambda env vars.
- Data flow: `Client → Function URL → Lambda invocation → LWA HTTP proxy → Express middleware → response`

## Cost

Region: eu-central-1. Workload: idle vs ~100K requests/month at 200ms avg duration, 512 MB.

| Resource | Idle | ~100K req/mo | Cost driver |
|---|---|---|---|
| Lambda requests | $0 | ~$0.02 | First 1M req/mo free |
| Lambda duration | $0 | ~$0.05 | GB-seconds (512 MB × 200ms × 100K) |
| ECR storage | ~$0.01 | ~$0.01 | ~100 MB image |
| Secrets Manager | $0.40 | $0.40 | $0.40/secret/mo; API calls negligible |
| CloudWatch Logs | $0 | ~$0.01 | 1-week retention |
| **Total** | **~$0.42** | **~$0.49** | Secrets Manager at idle, Lambda duration at scale |

Dominant cost driver: Lambda duration (scales with invocations, not time). At idle, cost is effectively $0 — Lambda scales to zero between requests.

## Notes

**Why Secrets Manager instead of SSM SecureString**

Other container patterns (ECS, App Runner) inject the API key from SSM SecureString — the platform resolves `{{resolve:ssm-secure:...}}` at task launch. Lambda does not support SSM SecureString dynamic references in environment variables ([CloudFormation docs](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/dynamic-references-ssm-secure-strings.html#dynamic-references-ssm-secure-strings-resources)). Secrets Manager references (`{{resolve:secretsmanager:...}}`) are supported, so this pattern uses a Secrets Manager secret instead. The trade-off is $0.40/mo per secret vs ~$0.05/mo for SSM SecureString.

**Lambda Web Adapter (LWA)**

LWA is an AWS-maintained binary that runs as a [Lambda extension](https://docs.aws.amazon.com/lambda/latest/dg/lambda-extensions.html). When Lambda starts a new execution environment:

1. Lambda detects the binary in `/opt/extensions/` and runs it before invoking the function handler
2. LWA starts the Express process (`node dist/server.js`) and polls `AWS_LWA_READINESS_CHECK_PATH` (`/health`) until it returns HTTP < 500
3. Once ready, LWA accepts Lambda invocations and translates them into HTTP requests to `localhost:3000`
4. The Express response is translated back into the Lambda response format

The app code requires zero changes — Express does not know it runs inside Lambda.

**Multi-stage Dockerfile**

The `example-container/Dockerfile` uses multi-stage targets to produce both the ECS and Lambda images from a single file. The `lambda` stage extends the shared `base` stage with three additions: the LWA binary copied into `/opt/extensions/`, and two `ENV` vars (`AWS_LWA_PORT=3000`, `AWS_LWA_READINESS_CHECK_PATH=/health`). See the [ECR README](../elastic-container-registry/README.md) for build and push commands.

**Cold start**

Container-based Lambda functions have cold starts of 1–3s (ARM64, 512 MB, ~100 MB image). A cold start occurs when Lambda needs a new execution environment — after a period of inactivity or to handle a new concurrent request. Subsequent invocations within the same environment are fast (<50ms overhead from LWA).

No provisioned concurrency is configured (the commented-out block in `stack.ts` shows how to add it). Enabling provisioned concurrency with 1 instance eliminates cold starts at a cost of ~$15/mo.

**Failure modes**

| Failure | Symptom | Mitigation |
|---|---|---|
| Cold start timeout | HTTP 500 (Lambda timeout); cold starts usually <3s but can spike | Increase `timeout` in `stack.ts` (max 15 min) |
| Bad deploy | Function returns 500/init error | No automatic rollback — push a known-good image with the `lambda` tag and `npx cdk deploy LambdaContainerStack` |
| Throttling | HTTP 429 from Function URL when >10 concurrent requests | Increase `reservedConcurrentExecutions` in `stack.ts`; or remove the cap for production |
| Pre-existing log group | CDK deploy fails with `ResourceAlreadyExistsException` | Delete the log group manually: `aws logs delete-log-group --log-group-name /aws/lambda/lambda-container` |

**Synchronous invocation model**

Function URL invocations are synchronous — Lambda executes, returns a response, done. There are no retries and no dead-letter queue. If the function times out or throws, the caller receives a 500/502 directly. This differs from async Lambda triggers (SQS, SNS, S3 events) where Lambda retries failures and can route them to a DLQ.

**Provisioned concurrency**

To eliminate cold starts, uncomment the `Alias` block in `stack.ts` and set `provisionedConcurrentExecutions: 1`. This keeps one execution environment warm at all times, at a cost of approximately $15/mo (1 instance × 512 MB × 730 hours). Provisioned concurrency cannot be set on `$LATEST` — it requires a function version and alias.

## Commands to play with stack

Prerequisites: ECR stack deployed, `:lambda` image pushed — see [ECR README](../elastic-container-registry/README.md). The API key is auto-generated by the stack via Secrets Manager.

```bash
# 1. Deploy
npx cdk deploy LambdaContainerStack

# 2. Get the Function URL (without / suffix)
FUNCTION_URL=$(aws cloudformation describe-stacks \
  --stack-name LambdaContainerStack \
  --query 'Stacks[0].Outputs[?OutputKey==`FunctionUrl`].OutputValue' \
  --output text | sed 's|/$||')

# 3. Health check (no auth required)
curl $FUNCTION_URL/health
# → {"status":"ok"}

# 4. Quote endpoint (API key required)
SECRET_ARN=$(aws cloudformation describe-stacks \
  --stack-name LambdaContainerStack \
  --query 'Stacks[0].Outputs[?OutputKey==`ApiKeySecretArn`].OutputValue' \
  --output text)
API_KEY=$(aws secretsmanager get-secret-value \
  --secret-id $SECRET_ARN \
  --query SecretString \
  --output text)

curl -H "x-api-key: $API_KEY" $FUNCTION_URL/quote
# → {"quote":"...","author":"..."}

# 5. Observe logs (streamed in real time)
aws logs tail /aws/lambda/lambda-container --follow

# 6. Synthesize CloudFormation template
npx cdk synth LambdaContainerStack > patterns/containers/lambda-container/cloud_formation.yaml

# 7. Destroy
npx cdk destroy LambdaContainerStack
```
